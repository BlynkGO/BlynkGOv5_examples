── FONT ─────────────────────────────────────────────────────────────────────────

  เมื่อ project ใช้ custom font ให้ output ครบทั้ง 3 อย่าง:

  ① download command  — คำสั่ง curl/wget สำหรับดาวน์โหลด .ttf จาก Google Fonts
  ② convert command   — คำสั่ง fontconv พร้อม flag ที่ถูกต้อง
  ③ FONT_DECLARE       — บรรทัด declare ใน main.cpp / CWC .h

  ✅ ตัวอย่าง output ที่ถูกต้อง:

    // ── [FONT] Kanit ──────────────────────────────────────────────────
    // ① Download (รันใน src/):
    //    curl -o Kanit.ttf "https://fonts.gstatic.com/s/kanit/v15/nKKZ-Go6G5tXcr4WORWz.ttf"
    //    หรือ: https://fonts.google.com/specimen/Kanit  → Download family
    // ② Convert (รันใน src/):
    //    fontconv --font Kanit.ttf --size 30 -thai -o Kanit_30.c
    //    fontconv --font Kanit.ttf --size 20 -thai -o Kanit_20.c
    // ─────────────────────────────────────────────────────────────────
    FONT_DECLARE(Kanit_30);
    FONT_DECLARE(Kanit_20);

  ⚠️  Google Fonts URL สำหรับ curl อาจเปลี่ยน version ได้
      ให้บอก URL หน้า specimen ด้วยเสมอเป็น fallback:
      https://fonts.google.com/specimen/<FontName>

── ICON ─────────────────────────────────────────────────────────────────────────

  เมื่อ project ใช้ custom icon ให้ output ครบทั้ง 3 อย่าง:

  ① iconlist.txt      — ไฟล์เนื้อหาพร้อม copy ไปวางใน src/ ได้เลย
  ② convert command   — คำสั่ง iconconv พร้อม flag
  ③ FONT_DECLARE + #define — บรรทัด declare และ symbol ทุกตัว

  ✅ ตัวอย่าง output ที่ถูกต้อง:

    // ── [ICON] ui_icons ──────────────────────────────────────────────
    // ① สร้างไฟล์ src/iconlist.txt ด้วยเนื้อหาต่อไปนี้:
    // ┌────────────────────────────────┐
    // │  thermostat      E846          │
    // │  humidity_mid    F176          │
    // │  air             E007          │
    // │  water_drop      E798          │
    // │  bolt            EA0B          │
    // └────────────────────────────────┘
    // ② Convert (รันใน src/):
    //    iconconv iconlist.txt --size 30 --name ui_icons
    // ③ iconconv จะ generate ให้อัตโนมัติ → copy ไปใส่ main.cpp:
    // ─────────────────────────────────────────────────────────────────
    #include "ui_icons_List.h"       // ← ไฟล์นี้ถูก generate โดย iconconv

    ภายใน ui_icons_List.h ตัวอย่างจะเป็นอย่างนี้ในอัตโนมัติ โดย iconconv
    ---------------------------
    #ifndef _UI_ICONS_LIST_H_
    #define _UI_ICONS_LIST_H_

    #define SYMBOL_THERMOSTAT   "\uE846"
    #define SYMBOL_HUMIDITY     "\uF176"
    #define SYMBOL_AIR          "\uE007"
    #define SYMBOL_WATER_DROP   "\uE798"
    #define SYMBOL_BOLT         "\uEA0B"

    #endif // _UI_ICONS_LIST_H_
    ---------------------------

  ⚠️  iconconv output pattern:
      - สร้างไฟล์ <name>_<size>.c   เช่น  ui_icons_30.c  ← font data (per size)
      - สร้างไฟล์ <name>_List.h     เช่น  ui_icons_List.h  ← SYMBOL_xxx defines
        → _List.h ใช้ร่วมกันได้ทุกขนาด  (SYMBOL codepoint ไม่เปลี่ยนตามขนาด)
        → ถ้าแปลงหลายขนาด ไฟล์ _List.h จะถูก overwrite แต่เนื้อหาเหมือนเดิม
      - ใน main.cpp: #include "_List.h" แค่ครั้งเดียว + FONT_DECLARE ทุกขนาดที่ใช้

  ตัวอย่างใช้หลายขนาด:
    #include "ui_icons_List.h"   // ← include ครั้งเดียว
    FONT_DECLARE(ui_icons_30);   // ← เพิ่ม FONT_DECLARE ทุกขนาดที่ต้องการ
    FONT_DECLARE(ui_icons_40);
    FONT_DECLARE(ui_icons_24);

── IMAGE ────────────────────────────────────────────────────────────────────────

  เมื่อ project ใช้ image ให้ output ครบทั้ง 3 อย่าง:

  ① download hint     — URL แหล่งภาพ + คำค้นที่แนะนำ (ภาพต้อง user เลือกเอง)
  ② convert command   — คำสั่ง imgconv พร้อม flag (-t ถ้าโปร่งใส)
  ③ IMAGE_DECLARE      — บรรทัด declare ใน main.cpp

  ✅ ตัวอย่าง output ที่ถูกต้อง:

    // ── [IMAGE] background ───────────────────────────────────────────
    // ① หาภาพ background (800×480):
    //    https://unsplash.com  → ค้น "dark green farm dashboard"
    //    Resize ให้เป็น 800×480 ก่อนแปลง
    // ② Convert (รันใน src/):
    //    imgconv background.png
    // ─────────────────────────────────────────────────────────────────
    IMAGE_DECLARE(img_background);

    // ── [IMAGE] logo (PNG โปร่งใส) ──────────────────────────────────
    // ① หาภาพ logo PNG โปร่งใส:
    //    https://flaticon.com  → ค้น "smart farm icon"
    // ② Convert (รันใน src/):
    //    imgconv logo.png -t      ← -t เพราะ PNG มี alpha
    // ─────────────────────────────────────────────────────────────────
    IMAGE_DECLARE(img_logo);

  ⚠️  image ต้อง user เลือกและดาวน์โหลดเอง — AI บอกได้แค่ URL แหล่งภาพ + คำค้น
      ไม่สามารถดาวน์โหลดภาพให้ได้โดยตรง

  ─ GRect ใช้เป็น layout container พร้อม background image ────
  GRect สามารถแสดง background image ได้โดยตรง
  โดยใช้ .image() ก่อน หรือพร้อมกับการตั้งค่า GRID/layout ได้เลย:

    rect.image(img_bg);
    rect.GRID_CELL( GRID_COL{...}, GRID_ROW{...} );
    rect.padding(30);
    // child widgets ใช้ .grid_cell() วางใน rect นี้ได้ทันที

  ✅ ใช้แทน GImage ได้เมื่อต้องการให้ container มี layout (GRID/padding)
  ✅ child ที่ใช้ .parent(rect) จะอยู่ในขอบเขต padding ของ rect อัตโนมัติ

EVENT CALLBACK:
  - ใช้ GWIDGET_CB macro เสมอ
  - ตัวอย่าง:
      btn.onClicked(GWIDGET_CB {
        // code here
      });

  - onClicked / onValueChanged: ถ้าต้องการ access CWC จาก callback
      ใช้ user_data (ส่ง this ตอน register) แล้วดึงผ่าน widget->user_data():
      btn.onClicked(GWIDGET_CB { ((MyClass*)widget->user_data())->doSomething(); }, this);

  - onSizeChanged: ใช้ดัก CWC ที่ต้อง recalc layout เมื่อขนาดเปลี่ยน
      ⚠️  ต้องเช็ค old_size vs new_size ก่อนเสมอ เพื่อป้องกัน refresh ที่ไม่จำเป็น

      กรณี 1 — CWC ดัก this ตัวเอง (cast widget ตรงๆ ได้เลย):
          myWidget.onSizeChanged(GWIDGET_CB {
            auto old_size = widget->event_old_size();
            auto new_size = widget->size();
            if (old_size.x != new_size.x || old_size.y != new_size.y) {
              MyClass* c = (MyClass*) widget;
              c->refresh_layout();
            }
          });

      กรณี 2 — ต้องส่ง external data เข้าไปด้วย:
          // ตั้ง user_data ก่อน register callback
          myWidget.user_data((void*)&myData);
          myWidget.onSizeChanged(GWIDGET_CB {
            auto old_size = widget->event_old_size();
            auto new_size = widget->size();
            if (old_size.x != new_size.x || old_size.y != new_size.y) {
              MyDataType* d = (MyDataType*) widget->user_data();
              // ใช้ d->...
            }
          });

      กรณีใช้งาน:
        - CWC ที่ใช้ SIZE_CONTENT + GRID_CELL แบบ fixed px → recalc GRID_COL/ROW ใหม่
        - รองรับ screen rotate runtime
        - ควร extract logic recalc ออกเป็น void refresh_layout() แยกต่างหาก
          แล้วเรียกทั้งจาก create() ครั้งแรก และจาก onSizeChanged

      ⚠️  RULE: register onSizeChanged เป็นบรรทัดสุดท้ายใน create() เสมอ
        - ต้องเรียก refresh_layout() ครั้งแรกก่อน แล้วค่อย register
        - ถ้า register ก่อน → BlynkGO trigger onSizeChanged ทันทีตอน GRect::create()
          → refresh_layout() ถูกเรียกขณะ child widgets ยังไม่ถูก create
          → GRID apply ผิด → child หลุดไปอยู่ผิดที่ (เช่น โผล่มุมบนซ้าย)

        // ✅ ลำดับที่ถูกต้องใน create():
        GRect::create();
        // ... setup child widgets, grid_cell, callbacks ...
        refresh_layout();                // ← ครั้งแรก ก่อน register เสมอ
        GRect::onSizeChanged(GWIDGET_CB{ // ← register หลังสุด
          auto old_size = widget->event_old_size();
          auto new_size = widget->size();
          if (old_size.x != new_size.x || old_size.y != new_size.y) {
            MyClass* c = (MyClass*) widget;
            c->_targetH = new_size.y;   // ← เก็บ size จริงไว้ใช้ใน refresh_layout()
            c->_targetW = new_size.x;   // ← ถ้าต้องการ width ด้วย
            c->refresh_layout();
          }
        });

      ⚠️  RULE: ใน refresh_layout() ห้ามอ่าน GRect::height()/width() โดยตรง
        - ถ้า CWC ใช้ SIZE_CONTENT → GRect::height() คืน constant ไม่ใช่ pixel จริง
        - ให้เก็บ size จริงไว้ใน private member _targetH / _targetW แทน
        - รับค่าจาก onSizeChanged → new_size.y / new_size.x
        - ครั้งแรก (ก่อน onSizeChanged fire) → fallback เป็น GScreen.height()

        // pattern ใน refresh_layout():
        int32_t h = (_targetH > 0) ? _targetH : GScreen.height();
        int32_t w = (_targetW > 0) ? _targetW : GScreen.width();

❌ ห้ามเรียก .create() เองกับ widget ปกติ — ไม่จำเป็นและไม่ใช่ pattern ที่ถูกต้อง

── PART 2: CUSTOM CLASS (สืบทอดจาก widget) ────────────────────────────────────

เมื่อสร้าง class ที่สืบทอดจาก widget ให้ใช้ LAZY CREATION PATTERN เสมอ:
  - constructor  → รับ parent เท่านั้น ห้ามตั้งค่าใดๆ
  - void create() → ตั้งค่าทั้งหมด เริ่มด้วย guard isCreated()
  - public methods → เรียก create() บรรทัดแรกเสมอ

❌ WRONG — ห้ามทำแบบนี้:
  class MyChart : public GChart {
    public:
      MyChart(GWidget* parent = nullptr) : GChart(parent) {
        this->set_size(600, 300);          // ❌ ยังไม่ถูก create!
        this->set_type(GCHART_LINE);       // ❌ method ไม่มีใน BlynkGO
        this->set_text_font(&prasanmit_25);// ❌ method ไม่มีใน BlynkGO
      }
  };

  MyChart my_chart(BlynkGO.container());  // ❌ ไม่มี BlynkGO.container()

  void setup() {
    BlynkGO.begin();
    my_chart.create();                     // ❌ ไม่ต้องเรียก create() เองจากข้างนอก
  }

✅ CORRECT — pattern มาตรฐาน BlynkGO:

  class MyChart : public GChart {
    public:
      MyChart(GPARENT) : GChart(parent) {}   // constructor รับ parent เท่านั้น

      void create() {
        if (GChart::isCreated()) return;      // guard: สร้างครั้งเดียว
        GChart::create();                     // สร้าง widget จริง ← ต้องมี

        GChart::size(600, 300);               // ตั้งค่าทั้งหมดใน create() เท่านั้น
        GChart::type(CHART_TYPE_LINE);        // ✅ CHART_TYPE_LINE ไม่ใช่ GCHART_LINE
        GChart::range(0, 100);
        GChart::point_count(100);
        GChart::bg_color(TFT_COLOR_HEX(0x1A1A1A));

        series = GChart::createSerie(TFT_CYAN);

        lb_title.parent(this);               // child widget ตั้งค่าใน create() ด้วย
        lb_title.font(prasanmit_25, TFT_WHITE);
        lb_title = "กราฟ";
      }

      void addValue(int32_t v) {
        create();                            // public method เรียก create() บรรทัดแรก
        GChart::addPoint(series, v);
      }

      chart_series_t* series;
      GLabel lb_title;
  };

  MyChart my_chart;        // ✅ global, ไม่ใช้ new, default parent = GScreen

  void setup() {
    BlynkGO.begin();
    my_chart.align_center(); // ✅ เรียก method ได้เลย — create() จะถูกเรียกอัตโนมัติ
                              //    ไม่ต้องเรียก my_chart.create() เองจาก setup()
  }

  ✅ widget ปกติ  → เรียก method หลัง BlynkGO.begin() ได้เลย ไม่ต้องเรียก create()
  ✅ custom class → create() จะถูก trigger อัตโนมัติตอนเรียก method แรก
  ✅ GPARENT macro → ใช้แทน (GWidget& parent = GScreen)
  ✅ default parent → GScreen (ไม่ต้องระบุถ้าต้องการให้อยู่บน GScreen)
  ❌ BlynkGO.container() → ไม่มีใน BlynkGO API
  ❌ new / pointer  → ไม่ใช้กับ BlynkGO widget
  ❌ GCHART_LINE    → ผิด ใช้ CHART_TYPE_LINE

    GRect::GRID_CELL( GRID_COL{ICON_W, GRID_FR(1), GRID_CONTENT},
                      GRID_ROW{GRID_FR(1), BAR_H} );
    lb_icon.grid_cell(0, 0, GRID_ALIGN_CENTER, GRID_ALIGN_CENTER);
    lb_title.grid_cell(1, 0, GRID_ALIGN_START,  GRID_ALIGN_CENTER);
    lb_value.grid_cell(2, 0, GRID_ALIGN_END,    GRID_ALIGN_CENTER);

  เหตุผล:
    ✅ content เปลี่ยน (font ใหม่, text ยาวขึ้น) → GRID reflow ให้อัตโนมัติ
    ✅ เพิ่ม/ลด child ได้โดยไม่ต้อง recalc offset ทุกตัว
    ✅ baseline/cap-top alignment ทำผ่าน offset_y ใน grid_cell() ได้ทันที
       (ดู Section 4e — ใส่ bl_offset / cap_offset ใน parameter สุดท้าย)
    ✅ padding() ของ card ควบคุม inner margin ที่เดียว ไม่ต้องชดเชยใน align()
    ❌ align() absolute — offset ผิดทันทีที่ font size หรือ card height เปลี่ยน

  เมื่อใช้ GRID_CELL ใน card-style CWC:
    - กำหนด GRID_CELL บน GRect (ตัว card เอง) ใน create()
    - child แต่ละตัวเรียก .grid_cell(col, row, ...) แทน .align(...)
    - ถ้ามี bar ให้วางเป็น row ล่างสุด span ทุก col:
        bar.grid_cell(0, 1, N_COLS, 1, GRID_ALIGN_STRETCH, GRID_ALIGN_CENTER);
    - padding() ของ card ทำหน้าที่เป็น inner margin ให้ทุก child พร้อมกัน

  ✅ ตัวอย่าง CWC สำหรับ SensorCard ที่ใช้ซ้ำ 4 ใบ (GRID layout):

CWC (Custom Widget Class) PATTERN — มี widget ลูก:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
class MyWidget : public GRect {
  public:
    GLabel lb_title;   // ① ประกาศเป็น value object เฉยๆ ห้ามทำอะไรในนี้

    // ② constructor: รับ GPARENT ส่งต่อให้แม่เท่านั้น
    //    ห้ามผูก parent ของลูก / ห้ามตั้งค่าอะไรทั้งสิ้น
    MyWidget(GPARENT) : GRect(parent) {}

    void create() {
      if (GRect::isCreated()) return;  // ③ guard
      GRect::create();                 // ④ สร้างแม่ก่อนเสมอ

      // ⑤ ผูก parent ลูกในนี้ (ผ่าน BlynkGO.begin() แล้ว ปลอดภัย)
      lb_title.parent(this);

      // ⑥ ตั้งค่า widget ลูก
      lb_title = "HELLO";
      lb_title.font(prasanmit_25);
    }

    // ⑦ public methods เรียก create() บรรทัดแรกเสมอ
    void title(String t) { create(); lb_title = t; }
};

กฎเหล็ก CWC:
  ❌ ห้ามผูก parent ลูกใน constructor
  ❌ ห้ามตั้งค่า widget ใดๆ ใน constructor
  ❌ ห้ามเรียก create() จากภายนอก (BlynkGO จัดการเอง)
  ✅ parent ลูกทุกตัวผูกใน create() เท่านั้น
  ✅ public method ทุกตัวเรียก create() บรรทัดแรกเสมอ

── CWC FILE RULE ────────────────────────────────────────────────────────────────

  ✅ CWC ทุกตัวต้องแยกเป็น .h ไฟล์ของตัวเอง  (ห้าม define ไว้ใน main.cpp)
  ✅ วางไฟล์ไว้ใน folder เดียวกับ main.cpp (src/)
  ✅ ใน main.cpp ให้ #include "CWC_Name.h" หลัง #include <BlynkGOv5.h>

  โครงสร้าง include ใน main.cpp:
    #include <BlynkGOv5.h>
    #include "CWC_SensorCard.h"
    #include "CWC_GGaugeGRAFANA.h"
    #include "CWC_StockCard.h"

── CWC NAMING CONVENTION ────────────────────────────────────────────────────────

  FILE  :  CWC_<ClassName>.h    ← prefix CWC_ เสมอ  (รู้ทันทีว่าเป็น CWC)
  CLASS :  <ClassName>          ← ไม่มี prefix, PascalCase, สื่อความหมาย

  ✅  CWC_SensorCard.h     → class SensorCard
  ✅  CWC_GGaugeGRAFANA.h  → class GGaugeGRAFANA
  ✅  CWC_StockCard.h      → class StockCard
  ✅  CWC_DonutChart.h     → class DonutChart
  ❌  SensorCard.h         ← ไม่รู้ว่าเป็น CWC หรือ helper
  ❌  CWCSensorCard.h      ← ไม่มี underscore อ่านยาก
  ❌  class CWCSensorCard  ← ห้าม prefix ใน class name

── GRID LAYOUT DIAGRAM RULE ─────────────────────────────────────────────────────

  CWC ทุกตัวที่มี internal GRID ต้องมี ASCII diagram ใน header comment เสมอ
  เพื่อให้ AI และ User เข้าใจ layout ก่อนอ่าน code

  กฎของ diagram:
    ✅ แสดง col/row ทุกช่อง พร้อมชื่อ content ใน [ ]
    ✅ บอก row size ที่ขอบขวา (GRID_FR(1) / px / GRID_CONTENT)
    ✅ บอก col size ด้านล่าง diagram
    ✅ แสดง GRID_COL{...} และ GRID_ROW{...} spec ตามหลัง diagram
    ✅ ถ้ามี span ให้เขียน "span N" หรือเส้น ─── ใน cell นั้น

  ตัวอย่าง diagram รูปแบบต่างๆ:

  // simple 2-col:
  //   ┌────────────┬────────────────────┐
  //   │ [icon]     │ [title / value]    │  ← row 0  GRID_FR(1)
  //   └────────────┴────────────────────┘
  //     col 0: 48px   col 1: GRID_FR(1)

  // 3-col + bar:
  //   ┌──────────┬───────────────┬──────────────┐
  //   │ [icon]   │ [title]       │ [value unit] │  ← row 0  GRID_FR(1)
  //   ├──────────┴───────────────┴──────────────┤
  //   │ [bar  ──────── span 3 ────────────────] │  ← row 1  4px
  //   └─────────────────────────────────────────┘
  //     col 0: 40px   col 1: GRID_FR(1)   col 2: GRID_CONTENT

  // title row + content row:
  //   ┌─────────────────────────────────────────┐
  //   │ [title ─────────────── span 3 ────────] │  ← row 0  GRID_CONTENT
  //   ├──────────┬───────────────┬──────────────┤
  //   │ [icon]   │ [value]       │ [unit]       │  ← row 1  GRID_FR(1)
  //   ├──────────┴───────────────┴──────────────┤

  SECTION 4d — SOFTDRAW: วาด text/shape โดยไม่สร้าง widget เพิ่ม
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

── แนวคิด ──────────────────────────────────────────────────────────────────────

  ใน CWC (หรือ widget ปกติ) สามารถวาด text / shape เพิ่มบนหน้า widget ได้โดยตรง
  ผ่าน draw callback โดยไม่ต้องสร้าง GLabel / GLine ฯลฯ เป็น member เพิ่ม
  → ประหยัด RAM และ widget object count ในกรณีที่ text เปลี่ยนตาม state เอง

  ใช้ได้ทั้งกับ widget ปกติ (setup()) และใน CWC::create()

── draw callback ที่ใช้ได้ ─────────────────────────────────────────────────────

  widget.onDrawMain(GWIDGET_CB { ... });
    → เรียกตอน BlynkGO กำลัง draw ส่วน GPART_MAIN ของ widget
    → วาดทับ bg/border ของ widget ได้เลย ยังไม่มี child อยู่ด้านบน

  widget.onDrawMainEnd(GWIDGET_CB { ... });
    → เรียกหลัง GPART_MAIN draw เสร็จ
    → ปลอดภัยกว่าสำหรับ text บน widget เพราะ child widget ยังไม่ render ทับ
    → แนะนำใช้กรณี draw label บน top-level widget

  widget.hookDrawTask(true);
  widget.onDrawTask(GWIDGET_CB { ... });
    → เรียกทุก GPART ที่ widget มี (KNOB, INDICATOR, MAIN ฯลฯ)
    → ใช้เมื่อต้องการ intercept การ draw ใน GPART เฉพาะ
    → ต้องเรียก hookDrawTask(true) ก่อน onDrawTask เสมอ
    → ใช้ DRAW_PART (macro) เพื่อเช็คว่า BlynkGO กำลัง draw GPART ไหน:
        if (DRAW_PART == DRAW_PART_KNOB) { ... }
        if (DRAW_PART == DRAW_PART_INDICATOR) { ... }
        if (DRAW_PART == DRAW_PART_MAIN) { ... }

  ⚠️  DRAW_PART, DRAW_AREA, DRAW_LAYER เป็น macro ที่ใช้ได้เฉพาะใน GWIDGET_CB
      ของ onDrawTask / onDrawMain / onDrawMainEnd เท่านั้น

── วิธีที่ 1: softdrawLabel() — shorthand สำหรับ text ────────────────────────

  ง่ายที่สุด ใช้ได้ใน onDrawMain / onDrawMainEnd
  วาด text บน widget โดยตรง ไม่ต้องสร้าง draw_label_dsc_t เอง

  // แบบ full options:
  widget.softdrawLabel(text, font, color, align, ofs_x, ofs_y);

  // แบบ shorthand (font/color default):
  widget.softdrawLabel(text, align, ofs_x, ofs_y);
  widget.softdrawLabel(text, color, align, ofs_x, ofs_y);

  ตัวอย่าง — วาดตัวเลขกลาง card โดยไม่ใช้ GLabel:

    class ValueCard : public GRect {
      public:
        ValueCard(GPARENT) : GRect(parent) {}
        void create() {
          if (GRect::isCreated()) return;
          GRect::create();
          GRect::bg_color(TFT_COLOR_HEX(0x1E1E1E));
          GRect::radius(12);
          GRect::height(80);
          GRect::onDrawMainEnd(GWIDGET_CB {
            ValueCard* c = (ValueCard*) widget;
            c->softdrawLabel(StringX::printf("%.1f", c->_val),
                             BebasNeueBold_num_45, TFT_WHITE,
                             ALIGN_CENTER, 0, 0);
            c->softdrawLabel(c->_unit,
                             prasanmit_20, TFT_COLOR_HEX(0x9CA3AF),
                             ALIGN_BOTTOM_MID, 0, -6);
          });
        }
        void value(float v) { create(); _val = v; GRect::invalidate(); }
        void unit(const char* u) { create(); _unit = u; GRect::invalidate(); }
      private:
        float  _val  = 0.0f;
        String _unit = "";
    };

  ⚠️  ต้องเรียก widget.invalidate() ทุกครั้งที่ข้อมูลเปลี่ยน
      เพื่อให้ BlynkGO trigger draw callback ใหม่

── วิธีที่ 2: draw_label_dsc_t + draw_label() — full control ───────────────────

  ใช้เมื่อต้องการควบคุม position ละเอียด เช่น จัด align ตาม area อื่น
  หรือต้องการ draw ใน layer แยก (draw_layer_create)

  ขั้นตอน:
    ① สร้าง draw_label_dsc_t และ init
    ② ตั้งค่า font, text, color
    ③ คำนวณ area ด้วย text_get_area()
    ④ จัด position ด้วย area_align()
    ⑤ วาดด้วย draw_label()

  ตัวอย่าง — ใน onDrawMainEnd บน GRect CWC:

    GRect::onDrawMainEnd(GWIDGET_CB {
      MyCard* c = (MyCard*) widget;
      draw_label_dsc_t label_dsc;
      area_t           label_area;
      draw_label_dsc_init(&label_dsc);

      // ── วาด title ──
      label_dsc.font  = &prasanmit_20;
      label_dsc.color = TFT_COLOR(TFT_COLOR_HEX(0x9CA3AF));
      label_dsc.text  = c->_title.c_str();
      text_get_area(&label_area, &label_dsc);
      area_align(DRAW_AREA, &label_area, ALIGN_TOP_LEFT, 12, 10);
      draw_label(DRAW_LAYER, &label_dsc, &label_area);

      // ── วาด value ──
      static String txt;
      txt = StringX::printf("%.1f", c->_val);
      label_dsc.font  = &BebasNeueBold_num_45;
      label_dsc.color = TFT_COLOR(TFT_WHITE);
      label_dsc.text  = txt.c_str();
      text_get_area(&label_area, &label_dsc);
      area_align(DRAW_AREA, &label_area, ALIGN_BOTTOM_RIGHT, -12, -8);
      draw_label(DRAW_LAYER, &label_dsc, &label_area);
    });

  helper functions:
    text_get_area(&area, &label_dsc)              ← คำนวณ bounding box ของ text
    text_get_area(&area, text, font)              ← overload แบบระบุ text+font ตรงๆ
    area_align(ref_area, &label_area, align, ox, oy) ← จัด label_area ให้ตรง ref
    area_align(DRAW_AREA, &label_area, align)     ← จัด relative to widget area
    draw_label(DRAW_LAYER, &label_dsc, &area)     ← วาดลง layer จริง

── วิธีที่ 3: onDrawTask — intercept GPART เฉพาะ ────────────────────────────

  SECTION 4e — BASELINE ALIGNMENT (จัด GLabel ต่างขนาด/ต่าง font บน baseline เดียวกัน)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

── ปัญหา ───────────────────────────────────────────────────────────────────────

  เมื่อวาง GLabel หลายตัวที่ใช้ font ต่างขนาด/ต่างชุดเรียงแนวนอน
  และใช้ GRID_ALIGN_CENTER หรือ align_center บน row/cell เดียวกัน
  ตัวอักษรจะไม่เรียงบน baseline เดียวกัน เพราะ:
    - GRID_ALIGN_CENTER จัด "กึ่งกลางของ font box" ให้ตรงกัน
    - แต่ baseline (เส้นฐานตัวอักษร) ของแต่ละ font อยู่ในตำแหน่งต่างกัน
      สัมพัทธ์กับกึ่งกลาง font box

  ผลที่เห็น: ตัวเลขใหญ่กับหน่วยเล็กลอยสูงต่ำไม่เรียงกัน น่าเกลียด

── หลักการคำนวณ offset ──────────────────────────────────────────────────────────

  ระยะจากกึ่งกลาง font box ลงมาถึง baseline (dist_below_center) ของ font ใดๆ:
    dist = (line_height - base_line) - line_height / 2

  เมื่อทั้งสอง label ใช้ GRID_ALIGN_CENTER บน cell เดียวกัน:
    กึ่งกลาง font box ของทั้งคู่ตรงกัน
    แต่ baseline ต่างกันเท่ากับ:
      bl_offset = dist_A - dist_B

  → ใส่ bl_offset นี้เป็น offset_y ของ label B ใน grid_cell()
    เพื่อดัน label B ให้ baseline ตรงกับ label A

── struct font_t ที่ใช้คำนวณ ────────────────────────────────────────────────────

  const font_t* f = lb.font();   // ดึง font pointer จาก GLabel
  f->line_height   // ความสูง bounding box ทั้งบรรทัด (px)
  f->base_line     // ระยะจาก baseline ขึ้นไปถึง bottom ของ bounding box (px)

── ตัวอย่าง — GRID cell เรียงแนวนอน ────────────────────────────────────────────

  สถานการณ์: lb_value ใช้ BebasNeueBold_num_45, lb_unit ใช้ prasanmit_25
  วางใน GRID col เคียงกัน ต้องการให้ baseline ตรงกัน

  // ① วาง lb_value ก่อน ใช้ GRID_ALIGN_CENTER เป็น anchor
  lb_value.grid_cell(2, 0, GRID_ALIGN_END, GRID_ALIGN_CENTER);

  // ② คำนวณ offset ของ lb_unit ให้ baseline ตรงกับ lb_value
  {
    const font_t* fv = lb_value.font();
    const font_t* fu = lb_unit.font();
    // dist_below_center = (line_height - base_line) - line_height/2
    int32_t dist_v  = (fv->line_height - fv->base_line) - fv->line_height / 2;
    int32_t dist_u  = (fu->line_height - fu->base_line) - fu->line_height / 2;
    int32_t bl_offset = dist_v - dist_u;   // ค่าบวก = ดัน lb_unit ลง, ลบ = ดันขึ้น
    lb_unit.grid_cell(3, 0, GRID_ALIGN_START, GRID_ALIGN_CENTER, 4, bl_offset);
  }

  // หมายเหตุ: bl_offset คำนวณได้ล่วงหน้าแล้วใส่ตัวเลขตรงๆ ก็ได้
  //   เช่น BebasNeueBold_num_45 vs prasanmit_25 → bl_offset ≈ +9

── ตัวอย่าง — align() เรียงแนวนอน (ไม่ใช้ GRID) ───────────────────────────────

  สถานการณ์: lb_big ใช้ font ขนาดใหญ่ เป็น anchor
             lb_small ใช้ font ขนาดเล็ก วางขวา lb_big ให้ baseline ตรงกัน

  lb_big.align_center();

  {
    const font_t* fb = lb_big.font();
    const font_t* fs = lb_small.font();
    int32_t dist_b  = (fb->line_height - fb->base_line) - fb->line_height / 2;
    int32_t dist_s  = (fs->line_height - fs->base_line) - fs->line_height / 2;
    int32_t bl_offset = dist_b - dist_s;
    // ALIGN_OUT_RIGHT_MID → กึ่งกลาง Y ตรงกัน, แล้ว shift ด้วย bl_offset
    lb_small.align(lb_big, ALIGN_OUT_RIGHT_MID, 4, bl_offset);
  }

── cap-top alignment — ด้านบนตัวอักษรตรงกัน ──────────────────────────────────────

  ใช้เมื่อต้องการให้ยอดด้านบนของตัวอักษร (cap-top / ascender) ตรงกัน
  แทนที่จะจัด baseline ให้ตรงกัน

  field ที่ใช้: f->cap_height
    = ระยะจาก baseline ขึ้นไปถึงยอดด้านบนของตัวพิมพ์ใหญ่ (เช่น A, H, T)
    (คล้าย base_line แต่ชี้ขึ้น)

  สูตร dist_above_center ของ font ใดๆ:
    dist = f->cap_height - f->line_height / 2
    (ค่าบวก = cap-top อยู่เหนือกึ่งกลาง font box)

  เมื่อทั้งสอง label ใช้ GRID_ALIGN_CENTER บน cell เดียวกัน:
    cap_offset = dist_A - dist_B
    → ใส่ -cap_offset เป็น offset_y ของ label B
      (ค่าลบ = ดัน B ขึ้น ให้ cap-top ตรงกับ A)

  ตัวอย่าง — GRID cell เรียงแนวนอน ต้องการให้ cap-top ตรงกัน:

    lb_value.grid_cell(2, 0, GRID_ALIGN_END, GRID_ALIGN_CENTER);

    {
      // Cap-top alignment:
      // dist_above_center = cap_height - line_height/2
      // cap_offset = dist_value - dist_unit
      // → ใส่ -cap_offset เป็น offset_y ของ lb_unit
      //   ค่าลบ = ดัน lb_unit ขึ้น ให้ cap-top ตรงกับ lb_value
      const font_t* fv = lb_value.font();
      const font_t* fu = lb_unit.font();
      int32_t dist_v   = fv->cap_height - fv->line_height / 2;
      int32_t dist_u   = fu->cap_height - fu->line_height / 2;
      int32_t cap_offset = dist_v - dist_u;
      lb_unit.grid_cell(3, 0, GRID_ALIGN_START, GRID_ALIGN_CENTER, 4, -cap_offset);
    }

  ตัวอย่าง — align() ไม่ใช้ GRID ต้องการให้ cap-top ตรงกัน:

    lb_big.align_center();
    {
      const font_t* fb = lb_big.font();
      const font_t* fs = lb_small.font();
      int32_t dist_b   = fb->cap_height - fb->line_height / 2;
      int32_t dist_s   = fs->cap_height - fs->line_height / 2;
      int32_t cap_offset = dist_b - dist_s;
      lb_small.align(lb_big, ALIGN_OUT_RIGHT_TOP, 4, -cap_offset);
    }

  ⚠️  สรุปความต่าง baseline vs cap-top:
    ┌─────────────────┬──────────────────────────────────────────────┬──────────────┐
    │ เป้าหมาย        │ สูตร dist                                    │ offset_y     │
    ├─────────────────┼──────────────────────────────────────────────┼──────────────┤
    │ baseline ตรงกัน │ (line_height - base_line) - line_height/2    │ +bl_offset   │
    │ cap-top ตรงกัน  │ cap_height - line_height/2                   │ -cap_offset  │
    └─────────────────┴──────────────────────────────────────────────┴──────────────┘
    ทั้งคู่ใช้สูตรเดียวกัน: offset = dist_A - dist_B
    ต่างกันที่ field และเครื่องหมาย offset_y ที่ใส่ใน grid_cell() / align()

── กฎที่ต้องทำตาม ───────────────────────────────────────────────────────────────

  ✅ เมื่อวาง GLabel ต่าง font เรียงแนวนอน (ค่า + หน่วย, ตัวเลข + label ฯลฯ)
     และต้องการให้ baseline ตรงกัน → ต้องคำนวณ bl_offset ทุกครั้ง
  ✅ เมื่อต้องการให้ cap-top ตรงกัน → ใช้สูตร cap_offset แทน
  ✅ ใช้สูตร dist = (line_height - base_line) - line_height/2  สำหรับ baseline
  ✅ ใช้สูตร dist = cap_height - line_height/2                  สำหรับ cap-top
  ✅ เปิด block {} กันตัวแปร dist ไม่ให้รั่วออกมา
  ✅ ใส่ comment อธิบายสูตรไว้ด้วย เพื่อให้อ่านแล้วเข้าใจได้ทันที

  ❌ ห้ามใช้ magic offset ตัวเลขลอยๆ โดยไม่มี comment อธิบาย
  ❌ ห้ามวาง label ต่าง font แนวนอนด้วย GRID_ALIGN_CENTER ล้วนๆ โดยไม่คำนวณ offset

ก่อน output code ให้ตรวจสอบทุกข้อ:

  [ ] BlynkGO.begin() อยู่ใน setup() ก่อน widget ทุกตัว
  [ ] BlynkGO.update() อยู่ใน loop()
  [ ] custom font ทุกตัวมี FONT_DECLARE() + comment [fontconv]
  [ ] icon ที่ไม่ใช่ built-in มี FONT_DECLARE() + iconlist.txt + comment [iconconv]
  [ ] image ทุกไฟล์มี IMAGE_DECLARE() + comment [imgconv]
  [ ] ใช้ wday() ไม่ใช่ weekday()
  [ ] event callback ใช้ GWIDGET_CB
  [ ] padding() เรียงลำดับ (top, bottom, left, right)
  [ ] ASSET PREPARATION comment block อยู่ด้านบนสุดของไฟล์
  [ ] widget ปกติ ไม่ต้องเรียก .create() เองจาก setup() — BlynkGO จัดการเอง
  [ ] custom class ทุกตัวใช้ GPARENT macro ใน constructor
  [ ] custom class ทุกตัวมี void create() พร้อม isCreated() guard
  [ ] custom class — public methods เรียก create() บรรทัดแรกเสมอ
  [ ] ไม่มีการตั้งค่า widget ใน constructor (ห้ามเด็ดขาด)
  [ ] ไม่มีการเรียก set_type(), set_text_font(), set_style_*() — ไม่มีใน BlynkGO
  [ ] GChart ใช้ type(), range(), point_count(), addPoint(), createSerie() เท่านั้น
  [ ] ใช้ CHART_TYPE_LINE ไม่ใช่ GCHART_LINE
  [ ] ไม่มีการใช้ new / pointer สำหรับ widget — declare เป็น value เสมอ
  [ ] ไม่มี BlynkGO.container() — default parent คือ GScreen ใส่อัตโนมัติ
  [ ] ทุก method ที่ใช้ปรากฏอยู่ใน example (02) หรือ reference (01) อย่างใดอย่างหนึ่ง
  [ ] ไม่มี method ที่ "เดา" มาเองโดยไม่มีหลักฐานจาก example หรือ reference
  [ ] ถ้า example กับ reference ขัดกัน → เชื่อ example เสมอ
  [ ] ถ้าไม่แน่ใจว่า method มีจริง → แจ้ง user แทนที่จะเดา
  [ ] widget ชนิดเดียวกัน config เหมือนกัน ซ้ำ > 1 ตัว → ทำ CWC แทน copy-paste
  [ ] CWC ทุกตัวใช้ LAZY CREATION PATTERN (constructor รับ parent เท่านั้น, create() มี guard)
  [ ] public method ของ CWC เรียก create() บรรทัดแรกเสมอ
  [ ] ชื่อ CWC สื่อความหมาย (SensorCard, StatBox, NavButton ฯลฯ)
  [ ] CWC ทุกตัวแยกเป็นไฟล์ CWC_<ClassName>.h — ห้าม define ไว้ใน main.cpp
  [ ] ชื่อไฟล์ขึ้นต้น CWC_ เสมอ, ชื่อ class ไม่มี prefix
  [ ] CWC_xxxxx.h ทุกไฟล์มี Store Header Block (DESCRIPTION / INTERNAL GRID LAYOUT / PUBLIC API / USAGE EXAMPLE)
  [ ] INTERNAL GRID LAYOUT ใน header มี ASCII diagram + GRID_COL/GRID_ROW spec ครบ
  [ ] Card-style CWC ใช้ GRID_CELL เป็น internal layout หลัก ไม่ใช่ align() absolute
  [ ] Card-style CWC — bar (ถ้ามี) วางเป็น row ล่างสุด span ทุก col ด้วย GRID_ALIGN_STRETCH
  [ ] text ที่ไม่ต้องการ interaction → พิจารณาใช้ softdrawLabel() / draw_label() แทน GLabel
  [ ] softdraw text เปลี่ยนค่า → เรียก invalidate() ทุกครั้งหลังอัปเดต state
  [ ] hookDrawTask(true) ต้องเรียกก่อน onDrawTask เสมอ
  [ ] draw_label_dsc_t.text ต้องชี้ไปยัง const char* ที่ยังมีชีวิตอยู่ขณะ draw (ใช้ static String)
  [ ] ไม่กำหนด size ให้ GRect โดยไม่จำเป็น — default PCT(100) อยู่แล้ว
  [ ] ไม่ระบุ GRID_ALIGN_STRETCH ใน grid_cell() ของ GRect — เป็น default อยู่แล้ว
  [ ] ไม่ใช้ position(0,0) เพื่อ reset ตำแหน่ง — default ทุก widget คือ ALIGN_CENTER
  [ ] ถ้าต้องการ background image + layout → ใช้ GRect.image() ไม่ใช่ GImage
  [ ] CWC ที่ใช้ onSizeChanged → register onSizeChanged เป็นบรรทัดสุดท้ายใน create() เสมอ
  [ ] CWC ที่ใช้ onSizeChanged → เรียก refresh_layout() ครั้งแรกก่อน register เสมอ
  [ ] refresh_layout() ห้ามอ่าน GRect::height()/width() โดยตรง — ใช้ _targetH/_targetW แทน
  [ ] _targetH/_targetW รับค่าจาก new_size ใน onSizeChanged, fallback เป็น GScreen.height()/width()
  [ ] GLabel ต่าง font เรียงแนวนอน → คำนวณ bl_offset (baseline) หรือ cap_offset (cap-top) ด้วยสูตร Section 4e
  [ ] ห้ามใช้ GRID_ALIGN_CENTER ล้วนๆ กับ label ต่าง font แนวนอน — baseline/cap-top จะไม่ตรงกัน
  [ ] ห้ามใส่ magic offset ตัวเลขลอยๆ โดยไม่มี comment อธิบายสูตร bl_offset / cap_offset
